مقدمه‌ای بر Django Admin

Django Admin چیست؟

Django Admin یک رابط مدیریتی خودکار (Auto-generated CRUD UI) است که روی مدل‌های ORM شما ساخته می‌شود. با چند خط کد در admin.py، لیست، جستجو، فیلتر، فرم ایجاد/ویرایش و حذف برای هر مدل در اختیار تیم داخلی قرار می‌گیرد.

مدل (models.py) ← ModelAdmin (admin.py) ← AdminSite ← URL /admin/ ← کاربر staff

درون‌کار چگونه عمل می‌کند؟

  1. ثبت مدل: admin.site.register(Model, ModelAdmin) مدل را به AdminSite پیش‌فرض وصل می‌کند.
  2. مسیریابی: django.contrib.admin برای هر مدل ثبت‌شده URLهای changelist، add، change و delete می‌سازد.
  3. ModelAdmin به‌عنوان کنترلر: کلاس ModelAdmin رفتار لیست، فرم، فیلتر، جستجو و دسترسی را تعریف می‌کند.
  4. فرم‌ها: Django از ModelForm بر اساس فیلدهای مدل فرم می‌سازد؛ شما با fieldsets و … آن را شکل می‌دهید.
  5. قالب‌ها: قالب‌های آماده در django/contrib/admin/templates رندر می‌شوند و قابل override هستند.

ثبت مدل‌ها (Registering Models)

admin.site.register

ساده‌ترین روش اتصال مدل به پنل:

from django.contrib import admin
from .models import Article

admin.site.register(Article)

در این حالت Django از ModelAdmin پیش‌فرض استفاده می‌کند — برای مدل‌های واقعی کافی نیست.

استفاده از ModelAdmin

@admin.register(Article)
class ArticleAdmin(admin.ModelAdmin):
    list_display = ("title", "status", "created_at")
    search_fields = ("title", "body")
    list_filter = ("status",)
نکته دکوراتور @admin.register خواناتر است؛ معادل آن admin.site.register(Article, ArticleAdmin) در انتهای فایل است.

بهترین روش‌ها

  • هر اپ یک فایل admin.py؛ مدل‌های مرتبط را گروه‌بندی کنید.
  • همیشه کلاس ModelAdmin اختصاصی با list_display و search_fields بنویسید.
  • برای اپ بزرگ، AdminSite جدا (multi-admin) در نظر بگیرید.

مبانی ModelAdmin

list_display

هدف: ستون‌های نمایش داده‌شده در صفحه لیست (changelist).

نحو: تاپل نام فیلدها یا متدهای نمایشی با دکوراتور @admin.display.

@admin.display(description="نام کامل", ordering="last_name")
def full_name(self, obj):
    return f"{obj.first_name} {obj.last_name}"

class PersonAdmin(admin.ModelAdmin):
    list_display = ("id", "full_name", "email", "is_active")

مثال واقعی: فروشگاه — ("sku", "title", "price", "stock", "is_published")

اشتباه رایج: قرار دادن فیلدهای ForeignKey بدون select_related → N+1 query.
list_display_links

هدف: مشخص کردن کدام ستون‌ها لینک به صفحه ویرایش هستند.

نحو: list_display_links = ("id", "title") — باید زیرمجموعه list_display باشد.

بهترین روش: یک یا دو ستون شناسایی (مثلاً عنوان یا کد) را لینک کنید؛ از لینک کردن همه ستون‌ها خودداری کنید.

list_editable

هدف: ویرایش سریع فیلدها مستقیماً از لیست، بدون ورود به فرم change.

مثال: list_editable = ("is_active", "sort_order")

محدودیت: نمی‌توان فیلدی را همزمان در list_display_links و list_editable داشت.

چه زمانی استفاده نکنیم: فیلدهای حساس (قیمت، نقش کاربر) یا فیلدهای با validation پیچیده.

ordering

هدف: ترتیب پیش‌فرض ردیف‌ها در changelist.

نحو: ordering = ("-created_at", "title") — منفی یعنی نزولی.

کاربر همچنان می‌تواند با کلیک روی سرستون مرتب‌سازی کند (اگر فیلد در list_display باشد).

list_per_page

هدف: تعداد ردیف در هر صفحه pagination.

پیش‌فرض: 100. برای UX بهتر معمولاً 25–50 مناسب است.

عملکرد: با LIMIT/OFFSET کار می‌کند؛ در صفحات انتهایی جداول بزرگ ممکن است کند شود (راه‌حل: cursor pagination سفارشی یا فیلتر اجباری).

empty_value_display

هدف: متن جایگزین وقتی مقدار فیلد خالی است (None یا رشته خالی).

مثال: empty_value_display = "—" یا "تعریف نشده"

خوانایی changelist را برای اپراتور بهتر می‌کند.

سیستم جستجو (Search)

search_fields

Django برای هر عبارت جستجو، شرط‌های OR با lookup پیش‌فرض icontains می‌سازد: WHERE field1 ILIKE '%term%' OR field2 ILIKE '%term%' ... 

class BookAdmin(admin.ModelAdmin):
    search_fields = (
        "title",
        "isbn",
        "author__name",      # جستجو در مدل مرتبط
        "author__email",
    )

جستجو در ForeignKey و مدل مرتبط

با syntax related_field__field به جدول join‌شده جستجو می‌کنید. Django JOIN می‌زند — بدون select_related در changelist همچنان برای نمایش هر ردیف ممکن است N+1 داشته باشید.

Lookupهای سفارشی در search_fields

search_fields = ("title", "slug", "id")  # id دقیق
# یا
search_fields = ("email__exact",)

عملکرد و ایندکس

موضوعتوصیه
icontains روی ستون بزرگFull Table Scan — از pg_trgm یا Elasticsearch برای جستجوی متنی استفاده کنید
جستجو روی idسریع (PK index)
فیلدهای پرجستجوdb_index=True یا ایندکس composite
چند JOIN در search_fieldsکوئری سنگین — محدود کردن فیلدها

نحوه ساخت کوئری جستجو در Django (داخلی)

در ModelAdmin.get_search_results(request, queryset, search_term)، Django برای هر کلمه در عبارت جستجو یک Q object می‌سازد: تمام فیلدهای search_fields با OR ترکیب می‌شوند؛ چند کلمه با AND بین آن‌ها. پیش‌فرض lookup: icontains (در PostgreSQL معادل ILIKE). 

def get_search_results(self, request, queryset, search_term):
    queryset, use_distinct = super().get_search_results(
        request, queryset, search_term
    )
    # مثال: محدود کردن جستجو به وضعیت فعال
    if search_term:
        queryset = queryset.filter(is_archived=False)
    return queryset, use_distinct

اگر جستجو روی M2M یا reverse FK باشد، use_distinct=True برمی‌گردد تا ردیف تکراری حذف شود.

اشتباه رایج جستجو روی TextField بدون محدودیت در جدول میلیونی — timeout قطعی است.

سیستم فیلتر (Filtering)

list_filter

فیلترهای سایدبار changelist. انواع:

  • BooleanField: بله/خیر/همه
  • DateField / DateTimeField: امروز، ۷ روز گذشته، این ماه، …
  • ForeignKey / Choice: لیست مقادیر مرتبط
  • RelatedOnlyFieldListFilter: فقط مقادیری که در داده موجودند
list_filter = (
    "is_active",
    "created_at",
    "category",
    ("author", admin.RelatedOnlyFieldListFilter),
)

SimpleListFilter — فیلتر سفارشی

class PublishedFilter(admin.SimpleListFilter):
    title = "وضعیت انتشار"
    parameter_name = "pub"

    def lookups(self, request, model_admin):
        return [("yes", "منتشر شده"), ("no", "پیش‌نویس")]

    def queryset(self, request, queryset):
        if self.value() == "yes":
            return queryset.filter(published_at__isnull=False)
        if self.value() == "no":
            return queryset.filter(published_at__isnull=True)
        return queryset
عملکرد هر فیلتر = یک پارامتر GET. ترکیب چند فیلتر می‌تواند ایندکس‌های composite را به چالش بکشد — ایندکس را با الگوی فیلترهای واقعی اپراتور طراحی کنید.

اینلاین‌ها (Inlines) — بخش تفصیلی

اینلاین چیست و چرا وجود دارد؟

وقتی دو مدل رابطه One-to-Many یا Many-to-Many دارند (مثلاً سفارش → اقلام سفارش)، ویرایش والد و فرزندان در یک صفحه منطقی است. InlineModelAdmin فرم‌های فرزند را داخل صفحه والد embed می‌کند.

Django از inlineformset_factory استفاده می‌کند — همان منطق Formset با ذخیره اتمی در یک transaction (در صورت پیکربندی).

مزایا

  • UX یکپارچه برای اپراتور
  • کمتر کلیک — ویرایش سفارش + اقلام همزمان
  • کد کمتر نسبت به viewهای جداگانه

معایب

  • صفحه شلوغ با فرزندان زیاد
  • خطر N+1 و فرم‌های سنگین
  • validation پیچیده بین والد و فرزند

TabularInline

هدف: نمایش فرزندان در قالب جدول (سطر = یک شیء فرزند).

کاربرد: فیلدهای کم تا متوسط، داده ساختاریافته — اقلام فاکتور، تصاویر محصول با چند ستون.

class OrderItemInline(admin.TabularInline):
    model = OrderItem
    extra = 0
    fields = ("product", "quantity", "unit_price")
    autocomplete_fields = ("product",)

@admin.register(Order)
class OrderAdmin(admin.ModelAdmin):
    inlines = [OrderItemInline]

StackedInline

هدف: هر فرزند در یک بلوک جدا با تمام فیلدها — شبیه فرم کوچک.

کاربرد: فیلدهای زیاد، نیاز به فضای عمودی — آدرس، مشخصات فنی، تنظیمات.

class LessonInline(admin.StackedInline):
    model = Lesson
    extra = 1
    fields = ("title", "video_url", "duration", "is_free")

اعتبارسنجی اینلاین

روش‌ها و سناریوهای رایج
  • سطح مدل: Model.clean() روی والد و فرزند — قبل از save
  • سطح فرم: InlineModelAdmin.form سفارشی با clean()
  • سطح formset: BaseInlineFormSet.clean() — مثلاً «حداقل یک قلم سفارش»
  • تعداد: min_num=1, validate_min=True
  • حذف: فرم‌های علامت‌خورده DELETE در formset — validation قبل از حذف نهایی
class OrderItemFormSet(BaseInlineFormSet):
    def clean(self):
        super().clean()
        if any(self.errors):
            return
        if not any(form.cleaned_data and not form.cleaned_data.get("DELETE")
                   for form in self.forms):
            raise ValidationError("حداقل یک قلم سفارش لازم است.")

عملکرد اینلاین

  • get_queryset روی Inline با select_related / prefetch_related
  • extra = 0 برای جلوگیری از فرم خالی اضافی
  • بیش از ~۲۰–۵۰ فرزند: paginate سفارشی یا صفحه جدا
  • raw_id_fields / autocomplete_fields برای FK فرزند

اکشن‌های سفارشی (Custom Actions)

اکشن‌ها توابعی هستند که روی queryset انتخاب‌شده در changelist اجرا می‌شوند. Django از چک‌باکس‌های کنار هر ردیف و دکمه «اجرا» استفاده می‌کند. مسیر داخلی: ModelAdmin.get_actions → اجرا در changelist_view.

اکشن ساده (Bulk Update)

@admin.action(description="انتشار موارد انتخاب‌شده")
def make_published(modeladmin, request, queryset):
    updated = queryset.update(status="published")
    modeladmin.message_user(request, f"{updated} مورد منتشر شد.", level=messages.SUCCESS)

class ArticleAdmin(admin.ModelAdmin):
    actions = [make_published]

queryset.update() یک کوئری SQL است — سریع. برای منطق پیچیده (سیگنال per-object) باید حلقه بزنید.

اکشن Export

خروجی CSV/Excel از ردیف‌های انتخاب‌شده. برای داده زیاد از صفحه میانی یا task پس‌زمینه (Celery) استفاده کنید تا timeout نگیرید.

اکشن با صفحه تأیید (Intermediate)

برای حذف دسته‌ای یا تغییر وضعیت حساس، کاربر را به view جدا هدایت کنید که لیست را نشان دهد و تأیید بگیرد — الگوی رسمی Django در مستندات Admin actions.

بهترین روشتوضیح
@admin.action(permissions=["..."])محدود به کاربران دارای permission
غیرفعال کردن اکشن پیش‌فرض حذفactions = None یا حذف selective
لاگ auditچه کسی چه bulk action زد
اشتباه رایج اجرای اکشن بدون محدودیت تعداد — انتخاب «همه» با select_across روی میلیون ردیف سرور را از کار می‌اندازد.

بهینه‌سازی کوئری (Query Optimization)

get_queryset

نقطه مرکزی کنترل داده در changelist، اکشن‌ها و اینلاین. هر جا queryset برمی‌گردد، فیلترهای کاربر و دسترسی را اعمال کنید. 

def get_queryset(self, request):
    qs = super().get_queryset(request)
    if not request.user.is_superuser:
        qs = qs.filter(organization=request.user.organization)
  return qs.select_related("author", "category").prefetch_related("tags")

select_related

برای ForeignKey و OneToOne — یک JOIN. در list_display هر بار که obj.author.name می‌خوانید بدون آن = یک query اضافه per row.

prefetch_related

برای ManyToMany و reverse FK — دو query (اصلی + مرتبط) به‌جای N. برای اینلاین حیاتی است.

annotate

from django.db.models import Count

def get_queryset(self, request):
    return super().get_queryset(request).annotate(_order_count=Count("orders"))

@admin.display(description="تعداد سفارش", ordering="_order_count")
def order_count(self, obj):
    return obj._order_count

ایندکس دیتابیس

فیلدهای list_filter، ordering و search_fields پرکاربرد را در مدل با db_index=True یا Meta.indexes علامت بزنید.

دسترسی‌ها (Permissions)

Django Admin از سیستم auth استفاده می‌کند: is_staff برای ورود به admin، و permissionهای app_label.action_modelname (مثلاً blog.change_article).

متدمعنی
has_module_permissionنمایش اپ در index admin
has_view_permissionمشاهده changelist/change (Django 2.1+)
has_add_permissionدکمه افزودن
has_change_permissionویرایش
has_delete_permissionحذف
 
def has_change_permission(self, request, obj=None):
    if not super().has_change_permission(request, obj):
        return False
    if obj is None:
        return True
    return obj.owner_id == request.user.id

با obj=None سطح changelist چک می‌شود؛ با شیء مشخص، سطح رکورد.

مقیاس‌پذیری Django Admin

حدود ۱۰٬۰۰۰ رکورد

Admin پیش‌فرض معمولاً کافی است. pagination + فیلتر + جستجوی ایندکس‌شده.

حدود ۱۰۰٬۰۰۰ رکورد

  • list_per_page = 50
  • autocomplete_fields برای همه FKهای سنگین
  • محدود کردن list_filter به فیلدهای selective
  • show_full_result_count = False — شمارش کل با COUNT(*) گران است

حدود ۱٬۰۰۰٬۰۰۰ رکورد و بیشتر

Admin به‌عنوان UI اصلی browse مناسب نیست مگر با معماری کمکی:

  • فیلتر اجباری (مثلاً تاریخ یا وضعیت) قبل از نمایش changelist
  • جستجوی full-text خارج از DB (Elasticsearch, PostgreSQL FTS)
  • Override get_search_results برای redirect به موتور جستجو
  • خواندن فقط — ویرایش از مسیرهای محدود
